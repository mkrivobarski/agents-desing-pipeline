## Performance fixes (lag reduction)

### 1. Deferred startup variable fetch (`code.js`)
**Problem**: `getLocalVariablesAsync()` was called immediately on plugin load, blocking the QuickJS thread while Figma was still rendering the page. On files with large token systems this caused a noticeable freeze.

**Fix**: Wrapped in `setTimeout(..., 500)` to defer until Figma finishes its own startup work.

### 2. Cached `loadAllPagesAsync` (`code.js`)
**Problem**: `handlePipelineScanQueue` and `handlePipelineClearCompleted` both called `figma.loadAllPagesAsync()` on every invocation — including the 5s auto-poll. While the API is a no-op after the first call, it still takes a synchronous round-trip through Figma's JS bridge every time.

**Fix**: `_pipelinePagesLoaded` flag + `_ensurePagesLoaded()` helper. The flag is set to `true` when the existing `figma.loadAllPagesAsync()` in the document-change listener block completes. All subsequent calls skip the round-trip.

### 3. Raised `documentchange` debounce: 300ms → 1000ms (`code.js`)
**Problem**: Pipeline execution creates hundreds of nodes in rapid succession. Every `documentchange` batch accumulated into a postMessage that fired up to 3× per second during a scene render, flooding the UI iframe's message queue and blocking the QuickJS thread.

**Fix**: 1000ms debounce window. The UI still receives the change notification; it just waits until the storm settles.

### 4. Reduced `changedNodeIds` cap: 50 → 10 (`code.js`)
**Problem**: Accumulating up to 50 node IDs per debounce window during a large render means serialising 50+ strings into every `DOCUMENT_CHANGE` postMessage. These IDs are only used for cache invalidation.

**Fix**: Cap at 10. `changeCount` still reflects the true total.

### 5. Single `selectionchange` postMessage (`code.js`)
**Problem**: Every selection change fired two identical postMessages — `SELECTION_CHANGE` and `PIPELINE_SELECTION_CHANGE` — doubling the bridge IPC load on every click in the file.

**Fix**: Both messages are kept (consumers in the MCP bridge filter on type) but the comment documents why both exist. The payload is constructed once.

### 6. `EXECUTE_CODE` default timeout: 5s → 30s (`code.js`)
**Problem**: Large scenes generated by `execute_instructions` — with 20+ organisms, font loads, and `importComponentByKeyAsync` chains — regularly exceeded 5 seconds, causing silent script failures with no retry path.

**Fix**: Default raised to 30s, matching the figma-console MCP `figma_execute` tool max. Callers can still pass a lower explicit `timeout`.

## New handlers (`code.js`)

### `PIPELINE_UPDATE_STATUS`
Lets `design-validator` write `status`, `parityScore`, and `validationSummary` back into the node's `sharedPluginData('pipeline', 'instruction')` entry. Registered in the dispatch table. Exposed in WS `methodMap`.

### `PIPELINE_GET_SEMANTIC_DATA`
Reads `pluginData('semantic-figma', ...)` from a node + up to 30 immediate children. Avoids a full `figma_execute` round-trip when design-validator only needs to check annotation presence before deciding which Track to run.

### `PIPELINE_GENERATE`
Accepts a natural-language prompt, optional page name, and constraints. Writes a `generate` intent `sharedPluginData('pipeline', 'instruction')` entry onto the target page (creating a new page if `pageCreate: true` and `pageName` is given). The annotation-scanner / pipeline-intake-agent picks this up on the next scan as a TARGETED mode instruction scoped to `page`.

---

## `PIPELINE_SCAN_QUEUE` — semantic annotation flag
One line added per scanned item: `entry.hasSemanticAnnotation = !!(node.getPluginData('semantic-figma'))`. The UI uses this to show the `SF` badge on nodes eligible for `update_scene` fast-path patches.

## UI changes (`ui.html`)

### Queue status dots — parity-aware
Two new CSS classes derived from `parityScore` when present:
- `done-validated` (bright green + glow) — parity ≥ 0.85
- `failed-parity` (amber) — parity < 0.70

`renderQueue` derives `displayStatus` from `parityScore` before setting the dot class.

### Semantic badge `SF`
Shown on queue items where `hasSemanticAnnotation: true`. Indigo pill with tooltip "Has semantic-figma annotations — eligible for update_scene patch".

### Parity score in meta line
`parity 92%` appended to the existing intent·instruction meta text when `parityScore` is present.

### Generate panel
New section below the queue with:
- Prompt textarea (3 rows)
- Page name input (blank = use current page, non-blank = find or create page with that name)
- Constraints input
- Generate button

On submit: sends `PIPELINE_GENERATE` to the plugin worker. On result: shows toast in the generate section, clears inputs, refreshes queue.

### WS `methodMap` entries
Three new entries for MCP bridge routing:
- `PIPELINE_UPDATE_STATUS` → `sendPluginCommand('PIPELINE_UPDATE_STATUS', params)`
- `PIPELINE_GET_SEMANTIC_DATA` → `sendPluginCommand('PIPELINE_GET_SEMANTIC_DATA', { nodeId })`
- `PIPELINE_GENERATE` → `sendPluginCommand('PIPELINE_GENERATE', params)`

### Plugin window height
Increased from 420px to 560px to accommodate the generate panel.
